# PokerClaw — 德州扑克 Agent 竞技平台 PRD

## 1. 产品概述

PokerClaw 是一个基于 LLM 的德州扑克 Agent 竞技平台。用户可以创建、配置并持久化 AI Agent，让它们在不同规则的德扑牌局中自主对战或与人类玩家竞技。平台提供完整的前端交互界面，支持牌局回放、Agent 思维过程展示、历史学习与 GTO 策略辅助。

---

## 2. 目标用户

| 角色 | 描述 |
|------|------|
| 扑克爱好者 | 希望与 AI 对战练习牌技，或观察不同策略风格的 AI 对局 |
| AI/LLM 研究者 | 希望测试不同 LLM 在不完全信息博弈中的决策能力 |
| 扑克教练 | 利用 Agent 回放和 GTO 分析进行教学 |
| 开发者 | 希望接入自定义 LLM 或扩展平台功能 |

## 3. 核心功能需求

### 3.1 牌局引擎

| ID | 功能 | 优先级 | 描述 |
|----|------|--------|------|
| F-ENG-01 | 无限注德州扑克规则引擎 | P0 | 完整实现 NLHE 规则：盲注、发牌、下注轮次（preflop/flop/turn/river）、摊牌、边池计算 |
| F-ENG-02 | 现金局模式 | P0 | 支持无限次买入(rebuy)，玩家可随时加入/离开 |
| F-ENG-03 | 锦标赛模式 | P1 | 单次买入或有限次买入，盲注递增，淘汰制 |
| F-ENG-04 | 桌面配置 | P0 | 可设置：玩家数(2-9)、初始筹码、大小盲注、ante |
| F-ENG-05 | 边池计算 | P0 | 当玩家 all-in 时正确计算主池和边池 |

### 3.2 Agent 系统

| ID | 功能 | 优先级 | 描述 |
|----|------|--------|------|
| F-AGT-01 | Agent 创建与配置 | P0 | 设置名称、水平(新手/中级/高手)、性格(松凶LAG/紧凶TAG/松弱calling station/紧弱rock/鱼fish) |
| F-AGT-02 | Agent 持久化存储 | P0 | Agent 配置、历史记录、学习数据持久存储于数据库 |
| F-AGT-03 | Agent 历史学习(短期) | P0 | 每次决策时将该 Agent 近期关键手牌摘要注入 LLM prompt |
| F-AGT-04 | Agent 历史学习(长期/RAG) | P1 | 将历史手牌向量化存储，决策时检索相似场景辅助决策 |
| F-AGT-05 | 多 LLM Provider 支持 | P0 | 可插拔 adapter 架构，每个 Agent 独立配置 LLM provider |
| F-AGT-06 | Agent 决策日志 | P0 | 记录每次决策的完整思考过程（LLM 原始输出） |
| F-AGT-07 | Agent 性格一致性 | P0 | 通过 system prompt 工程确保 Agent 行为符合设定的性格特征 |

### 3.3 人类玩家

| ID | 功能 | 优先级 | 描述 |
|----|------|--------|------|
| F-HMN-01 | 实时参与对战 | P0 | 人类通过 Web 页面实时看牌、下注、弃牌等 |
| F-HMN-02 | 观战模式 | P0 | 人类可观看正在进行的 Agent 对局 |
| F-HMN-03 | 操作计时 | P1 | 人类玩家有操作时间限制，超时自动弃牌 |

### 3.4 牌局回放与历史

| ID | 功能 | 优先级 | 描述 |
|----|------|--------|------|
| F-RPL-01 | 整局回放 | P0 | 可回放某一完整竞技(session)的所有手牌 |
| F-RPL-02 | 单手回放 | P0 | 可回放某一手牌的完整过程 |
| F-RPL-03 | 手牌信息展示 | P0 | 回放中展示：各玩家手牌、公共牌、下注历史 |
| F-RPL-04 | Agent 思维展示 | P0 | 回放中展示 Agent 的决策逻辑和思考过程 |
| F-RPL-05 | 按玩家/Agent 筛选 | P0 | 支持按特定 Agent 或玩家粒度回溯牌局历史 |
| F-RPL-06 | 回放控制 | P1 | 播放/暂停/快进/后退/跳转到特定街 |

### 3.5 手牌实验室 (Hand Lab)

| ID | 功能 | 优先级 | 描述 |
|----|------|--------|------|
| F-LAB-01 | 预设场景创建 | P0 | 选择 Agent，为每个 Agent 指定手牌（或留空随机发牌）和筹码数量 |
| F-LAB-02 | 公共牌预设 | P0 | 可预设 flop/turn/river 的公共牌，或留空随机发牌 |
| F-LAB-03 | 起始街设置 | P1 | 可选择从 preflop/flop/turn/river 任一街开始 |
| F-LAB-04 | 场景运行 | P0 | 运行预设场景，观察各 Agent 在给定条件下的实际决策 |
| F-LAB-05 | 同场景多次运行 | P1 | 同一预设场景可重复运行多次，观察 Agent 决策的稳定性和差异 |
| F-LAB-06 | 场景保存与加载 | P1 | 将预设场景保存为模板，后续可加载复用 |
| F-LAB-07 | 实验结果对比 | P2 | 同一场景不同 Agent 组合的结果对比分析 |

### 3.6 GTO 辅助

| ID | 功能 | 优先级 | 描述 |
|----|------|--------|------|
| F-GTO-01 | GTO 策略参考 | P1 | Agent 决策时可参考 GTO 策略建议 |
| F-GTO-02 | GTO 模式 Agent | P2 | 可创建严格按 GTO 策略行动的 Agent |
| F-GTO-03 | 手牌 GTO 分析 | P2 | 回放时可查看某一决策点的 GTO 推荐 |

### 3.7 监控与可观测性

| ID | 功能 | 优先级 | 描述 |
|----|------|--------|------|
| F-MON-01 | LLM 调用 Token 统计 | P0 | 记录每次 Agent 决策的 input_tokens、output_tokens、总 token 数 |
| F-MON-02 | LLM 调用耗时 | P0 | 记录每次 LLM 调用的请求耗时(ms)，含网络延迟 |
| F-MON-03 | Agent 决策成功率 | P0 | 统计 Agent 成功决策 vs 超时/解析失败/异常的比率 |
| F-MON-04 | LLM 接口稳定性 | P0 | 记录每次 LLM 调用的成功/失败/超时状态，统计各 Provider 的可用率 |
| F-MON-05 | Agent 超时兜底 | P0 | Agent 30 秒内未返回决策，自动视为弃牌(fold)，并记录超时事件 |
| F-MON-06 | Session 级汇总统计 | P1 | 每个 session 的总 token 消耗、总耗时、总费用估算 |
| F-MON-07 | Agent 级汇总统计 | P0 | 每个 Agent 的历史总 token、平均决策耗时、成功率、超时次数 |
| F-MON-08 | 监控仪表盘 | P1 | 前端页面展示实时和历史监控数据，支持按 Agent/Session/Provider 筛选 |
| F-MON-09 | 异常告警提示 | P2 | 当 Agent 连续超时或 Provider 错误率过高时，前端弹出告警 |

## 4. 非功能需求

| 类别 | 要求 |
|------|------|
| 代码规范 | 变量命名体现含义；关键逻辑有注释；单个 .py 文件 ≤ 300 行 |
| 测试 | 每个子函数有对应测试文件；支持最小原型试跑 |
| 可扩展性 | LLM adapter 可插拔；规则引擎可扩展新变体 |
| 性能 | 支持同时运行至少 2 桌牌局；Agent 决策延迟主要受 LLM API 限制 |
| 容错 | Agent 决策超时(30s) 自动 fold；LLM 调用失败自动重试 1 次后降级 fold |
| 可观测性 | 每次 LLM 调用记录 token 数、耗时、状态；提供 Agent 和 Session 级汇总统计 |
| 数据持久化 | SQLite 存储；架构上预留切换 PostgreSQL 的能力 |

| 层 | 技术 |

## 6. 用户故事

### US-01: Agent 自主对战
> 作为用户，我想配置 4 个不同性格的 Agent（1个高手TAG、1个高手LAG、1个新手鱼、1个中级calling station），让它们在一个现金局中对战 100 手牌，然后回放精彩手牌并查看每个 Agent 的思考过程。

### US-02: 人机对战
> 作为用户，我想加入一个有 3 个 Agent 的牌桌，亲自参与对战。我能实时看到自己的手牌，在限时内做出下注决策。

### US-03: Agent 学习进步
> 作为用户，我创建了一个"小白"Agent，让它打了 500 手牌。之后再让它打，它应该能基于历史经验做出更好的决策（比如识别出类似场景下之前的失误）。

### US-04: 锦标赛模式
> 作为用户，我想组织一场 6 人锦标赛（5 Agent + 我），每人 1000 筹码单次买入，盲注每 10 手递增一次，直到决出冠军。

### US-05: 不同 LLM 对比
> 作为用户，我想创建 3 个 Agent 分别使用 Claude、GPT-4、DeepSeek，同样设置为"高手TAG"，让它们对战观察哪个 LLM 更擅长打牌。

### US-06: 手牌实验室复盘
> 作为用户，我想设置一个特定场景：4 个 Agent 在翻牌圈，Agent-A 拿 A♠K♠（2000筹码），Agent-B 拿 Q♥Q♦（1500筹码），另外 2 个 Agent 手牌随机。公共牌设为 K♥Q♠9♠。我想观察各 Agent 在这个特定局面下如何决策，并且重复运行 5 次看决策是否一致。

### US-07: 教学场景分析
> 作为用户，我想保存一组经典牌局场景（如"湿润面的同花听牌对顶对"），用不同性格的 Agent 去跑这些场景，对比高手TAG和新手鱼在相同场景下的不同决策，用于教学。

## 7. 约束与假设

### 约束
- Agent 决策速度受 LLM API 响应速度限制
- GTO solver 运算密集，可能需要预计算或缓存常见场景
- 单人开发/小团队，需控制复杂度

### 假设
- 用户自行提供 LLM API Key
- 初期不需要用户认证系统（单用户使用）
- 牌局并发量低（≤5 桌同时进行）

## 8. 成功指标

| 指标 | 目标 |
|------|------|
| 引擎正确性 | 所有规则边界 case 通过测试（边池、all-in、split pot 等） |
| Agent 可用性 | Agent 能在 30 秒内完成一次决策 |
| 回放完整性 | 任意手牌可完整回放所有信息 |
| 学习有效性 | 经过 100+ 手牌训练的 Agent 在同等对手面前胜率有可观测提升 |
